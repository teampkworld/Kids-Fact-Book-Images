1) ## Fix Book Spine Title Editor – Ensure Preview and PDF Render Identically

### Current Issue

The **Book Spine Title Editor** is rendering the spine title differently in the **live preview** and the **exported PDF**.

For example:

* The font size displayed in the preview does not match the font size in the generated PDF.
* The title position, spacing, or scaling may also differ between the preview and the PDF.

This creates an inaccurate preview, making it impossible for the user to know how the final printed cover will appear.

---

## Required Fix

Implement a **single rendering engine** for both the **live preview** and the **PDF export** so that they use the exact same rendering logic, measurements, and calculations.

The live preview must be a true **WYSIWYG (What You See Is What You Get)** representation of the final PDF.

---

## Rendering Requirements

Both the preview and the PDF must use the same:

* Font family
* Font size
* Font weight (Bold)
* Italic
* Underline
* Letter spacing
* Line spacing
* Text color
* Background color
* Text rotation
* Text alignment
* Text positioning
* Spine width calculation
* DPI and scaling calculations
* Font metrics and baseline calculations

No separate calculations should be used for the preview and the PDF.

---

## Rendering Logic

Create a shared rendering pipeline that is used by both the preview and the PDF export.

The workflow should be:

1. Read the user's spine title settings.
2. Calculate the available spine area.
3. Apply font metrics and typography settings.
4. Calculate the final text size and position.
5. Render the title using the same layout engine.
6. Display the result in the live preview.
7. Reuse the exact same rendering data when generating the PDF.

The PDF must not recalculate or modify the layout independently.

---

## Live Preview Accuracy

The preview should display **exactly** what will be exported.

If the user selects:

* 6 pt font
* Arial
* Bold
* Letter Spacing = 8
* Line Spacing = 1.2
* Black background
* White text

the preview must look **identical** to the final PDF.

No differences in size, spacing, alignment, or positioning are acceptable.

---

## Automatic Updates

Whenever the user changes any spine setting, automatically:

* Recalculate the spine layout.
* Update the live preview immediately.
* Store the final rendering values.
* Use those exact values during PDF generation.

No manual refresh or regeneration should be required.

---

## Validation Checklist

Before generating the final KDP cover PDF, verify that:

* ✅ The spine title has the same font size in both the preview and the PDF.
* ✅ Font family, style, and formatting match exactly.
* ✅ Letter spacing and line spacing are identical.
* ✅ Text position and alignment are identical.
* ✅ Spine width calculations are identical.
* ✅ The preview is a true WYSIWYG representation of the exported PDF.
* ✅ The exported PDF matches the preview pixel-for-pixel, with no differences in typography, scaling, or layout.

2) ## Add Gradient Background Presets for the Book Spine

### Required Implementation

In the **Book Spine Title Editor**, add a new section called **Spine Background Style** that allows users to apply professional gradient backgrounds to the book spine.

---

### Spine Background Options

Provide the following background options:

* Solid Color
* Transparent
* Image Upload
* **Gradient Presets** (New)

When **Gradient Presets** is selected, display a gallery of professionally designed gradients that can be applied to the spine with a single click.

* Blue Gradient
