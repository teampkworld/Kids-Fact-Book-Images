#### 2. Fix SVG Upload Replacement & Cache Logic

There is a cache/update issue in the **"MAP OF YOUR ADVENTURE" Bulk SVG Upload (Index)** section.

**Current Problems**

* When the user uploads a new set of transparent SVG icons, the previously uploaded icons are still being used.
* Old SVG files are not being deleted or replaced.
* If the user uploads only some icons and leaves other upload slots empty, those empty slots continue displaying previously uploaded icons instead of remaining empty.
* The PDF output therefore contains outdated icons that should no longer exist.

**Required Logic**

Implement a complete replacement workflow instead of merging uploads.

* Every new upload session must completely replace the previous icon set.
* Before importing new SVG files:

  * Delete all previously stored SVG files.
  * Clear all in-memory icon mappings.
  * Remove all cached SVG references.
  * Reset every upload slot to an empty state.
* Then import and register only the newly uploaded SVG files.
* If an upload slot is left empty, it must remain empty. Never reuse an icon from a previous upload.
* The PDF generator must always use the current upload session only and must never reference cached or previously uploaded SVG files.
* Ensure the upload preview, internal storage, icon mapping, and final PDF all stay synchronized with the latest uploaded files.
* If the user uploads another new icon set later, repeat the same process:

  1. Completely delete the previous icon set.
  2. Clear all caches and mappings.
  3. Import the new SVG files.
  4. Generate the PDF using only the newly uploaded icons.

This logic must guarantee that no outdated SVG icons can appear anywhere in the application or in the generated PDF.
