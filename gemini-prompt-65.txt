# Update Table of Contents SVG Icons and Implement Automatic Asset Refresh

## Current Problem

The **Table of Contents** page is displaying **outdated SVG icons** for the following headings:

* **MY FAVORITE FACTS**
* **FACT EXPLORER CHALLENGE**
* **AMAZING FACT COLLECTOR**

The old SVG files have already been **removed and replaced** in the GitHub repository with **new SVG files using the same filenames and the same URLs**.

However, the application is still displaying the **old cached SVG icons** instead of the updated versions.

---

# Required Fix

Update the SVG loading logic so that the application always displays the **latest version** of the SVG icons from the GitHub repository.

Do not continue using cached or previously downloaded SVG assets.

The following headings must display their corresponding **new SVG icons**:

* **MY FAVORITE FACTS**
* **FACT EXPLORER CHALLENGE**
* **AMAZING FACT COLLECTOR**

---

# Root Cause to Investigate

The most likely issue is that the application is using one or more of the following:

* Browser cache
* Service Worker cache
* Local storage
* Memory cache
* Static asset cache
* Previously generated PDF cache
* Build cache
* Image preloading cache

Because the filename and URL remain unchanged, the application continues rendering the old cached SVG instead of requesting the updated file.

---

# Required Implementation

When rendering the Table of Contents:

1. Always fetch the latest SVG from GitHub.
2. Detect if the remote SVG has changed.
3. Invalidate any cached copy.
4. Replace the cached SVG with the newest version.
5. Render only the latest SVG in both the application preview and the generated PDF.

---

# Automatic Asset Update Logic

Implement automatic asset synchronization so the application checks for updated SVG files whenever it loads or generates a new book.

The system should:

* Detect if the SVG file has been modified.
* Refresh the asset automatically.
* Replace the previous version.
* Avoid requiring manual code changes whenever SVG artwork is updated.

If the SVG file is updated in GitHub while keeping the same filename and URL, the application must automatically use the updated version.

---

# PDF Generation

The generated PDF must always embed the **latest SVG icons**, not previously cached versions.

Verify that:

* The preview uses the latest SVG.
* The exported PDF uses the same latest SVG.
* The SVG is embedded correctly without falling back to an older cached copy.

---

# Validation Checklist

Before generating the final PDF, verify that:

* ✅ **MY FAVORITE FACTS** displays the latest SVG icon.
* ✅ **FACT EXPLORER CHALLENGE** displays the latest SVG icon.
* ✅ **AMAZING FACT COLLECTOR** displays the latest SVG icon.
* ✅ No old SVG assets are displayed.
* ✅ Browser, application, and PDF caches are invalidated when SVG files are updated.
* ✅ The preview and PDF always use the same, most recent SVG files.
* ✅ Future SVG updates with the same filename and URL are detected and reflected automatically without requiring manual intervention.

---

# Success Criteria

Whenever an SVG icon is updated in the GitHub repository—even if the **filename and URL remain unchanged**—the application must automatically detect the change, refresh the cached asset, and display the **new SVG icon** in both the Table of Contents and the generated PDF. Old icons must never continue to appear after an update.
